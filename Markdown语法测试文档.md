# Markdown 语法全面测试文档

> 本文档用于测试 **MiniMarkdown** 的渲染与编辑能力。它尽可能涵盖了从最基础到较为冷门的各类 Markdown 语法，既可以在「对照模式」中查看源码与预览（基于 `marked`）的差异，也可以在「所见即所得模式」中检验 TipTap 编辑器对各类节点的支持程度。建议你逐节对照，留意哪些语法被正确渲染、哪些被忽略或显示为纯文本。

本文约一万字，混合了中文、英文与代码，用来同时检验中西文混排、标点挤压、等宽字体以及长段落的换行表现。阅读时不妨重点关注以下几个维度：标题层级是否清晰、列表缩进是否准确、代码块是否高亮且保留空白、表格是否对齐、引用是否有层次、以及图片与链接能否正常加载。

## 一、文档元信息与说明

这一节是纯段落文本，用来测试基础排版。一个段落由连续的若干行组成，行与行之间如果**没有空行**，通常会被渲染器合并为同一段；只有当中间出现**一个空行**时，才会真正开始一个新的段落。下面这三行在源码里是分开写的，但在预览中应当显示为同一段连续文字：
这是第一行，
这是第二行，
这是第三行。

而下面这一段与上面之间空了一行，因此它是一个**独立的段落**。段落是 Markdown 中最基本的块级元素，所有行内语法（粗体、斜体、链接等）都可以嵌入到段落之中。

## 二、标题层级（H1–H6）

标题用于组织文档结构，左侧大纲面板会根据标题自动生成目录。Markdown 支持六级标题，使用井号 `#` 的数量来表示层级：

# 一级标题 Heading 1
## 二级标题 Heading 2
### 三级标题 Heading 3
#### 四级标题 Heading 4
##### 五级标题 Heading 5
###### 六级标题 Heading 6

除了井号写法，一级与二级标题还支持「Setext」下划线写法，用等号或连字符标在文字下方：

另一种一级标题
=============

另一种二级标题
-------------

标题里同样可以包含行内格式，例如 ### 含有 `代码` 与 **粗体** 的标题，用来检验标题内的混合渲染是否正常。

## 三、文本强调与行内格式

行内格式是日常写作中使用最频繁的语法。下面集中演示各种强调方式：

- 这是 **使用双星号的粗体**，也可以写成 __双下划线的粗体__。
- 这是 *使用单星号的斜体*，也可以写成 _单下划线的斜体_。
- 这是 ***三星号的粗斜体***，同时具有加粗与倾斜效果。
- 这是 ~~删除线文本~~，用于表示废弃或更正的内容（GFM 扩展语法）。
- 这是 `行内代码`，等宽字体显示，适合标注变量名 `userId`、函数 `flushSave()` 或路径 `src/App.tsx`。
- 行内代码里如果需要显示反引号本身，可以用双反引号包裹：`` 这里有一个反引号 ` 字符 ``。

强调可以相互嵌套，例如：**加粗中包含 *斜体* 的情况**，以及 *斜体中包含 `代码` 的情况*。还可以测试中英文之间的强调边界，比如「中文**English混排**中文」是否能正确加粗中间的英文。

部分编辑器还支持下列扩展强调（不一定被支持，正好用来测试）：H~2~O 中的下标、x^2^ 中的上标、以及 ==高亮文本== 的标记语法。

function debounce(fn, delay = 300) {
  let timer = null;
  return function (...args) {

  };
}

## 五、引用块

引用块用于引述他人观点或突出提示信息。最简单的引用只需在行首加 `>`：

> 简洁是可靠的前提。 —— Edsger W. Dijkstra

引用块可以包含多个段落，只要每个段落行首都有 `>`，并用一个含 `>` 的空行分隔：

> 第一段引用内容。这里可以写较长的文字，用来测试引用块在换行时是否保留左侧的竖线样式。
>
> 第二段引用内容，与上一段同属一个引用块。

引用还可以**多层嵌套**，每多一层就多一个 `>`：

> 第一层引用。
>
> > 第二层引用（被第一层包裹）。
> >
> > > 第三层引用，层层递进。

引用块内部也能容纳其它块级元素，例如列表、代码和强调：

> **重点提示：**
>
> 1. 第一步：安装依赖 `pnpm install`；
> 2. 第二步：启动 `pnpm tauri dev`；
>
> > 注意：首次编译 Rust 可能较慢，请耐心等待。
>
> ```bash
> pnpm tauri build
> ```

## 七、有序列表

有序列表用「数字加点」表示。值得注意的是，渲染器通常只看第一个序号，后续即使全写 `1.` 也会自动递增：

1. 准备阶段
2. 开发阶段
3. 测试阶段
4. 发布阶段

下面这种「全部写 1.」的写法在预览中也应自动编号为 1、2、3：

1. 第一步
1. 第二步
1. 第三步

有序列表同样支持嵌套，并可与无序列表混合：

1. 后端任务
   1. 设计数据库表
   2. 编写接口
      - 鉴权接口
      - 文档接口
   3. 编写测试
2. 前端任务
   1. 搭建页面框架
   2. 对接接口

部分扩展语法支持自定义起始序号，例如从 5 开始：

5. 第五项
6. 第六项
7. 第七项

## 八、任务列表（GFM）

任务列表（待办清单）是 GitHub 风格的扩展语法，用 `- [ ]` 表示未完成、`- [x]` 表示已完成。这是检验渲染器是否支持 GFM 的好例子：

- [x] 修复窗口拖动问题
- [x] 工具条与控件对齐
- [x] 授予 `core:window:allow-start-dragging` 权限
- [ ] 编写更多单元测试
- [ ] 支持图片粘贴上传
  - [x] 子任务：调研可行方案
  - [ ] 子任务：实现上传逻辑

## 九、表格（GFM）

表格使用竖线 `|` 分隔单元格，第二行用连字符定义表头分隔线，并可通过冒号控制对齐方式（左对齐、居中、右对齐）。

| 语法 | 写法示例 | 是否常用 | 备注 |
| :--- | :--- | :---: | ---: |
| 粗体 | `**文字**` | 是 | 行内 |
| 斜体 | `*文字*` | 是 | 行内 |
| 代码块 | ```` ```js ```` | 是 | 块级 |
| 表格 | `\| a \| b \|` | 中 | GFM |
| 脚注 | `[^1]` | 低 | 扩展 |

表格单元格内也可以包含行内格式：

| 功能 | 状态 | 负责人 |
| --- | --- | --- |
| **拖动修复** | ✅ 已完成 | `@dev` |
| *布局调整* | ✅ 已完成 | `@dev` |
| ~~旧打包版~~ | 🗑️ 已删除 | — |

## 十三、转义字符

当你想显示 Markdown 的特殊符号本身、而不希望它被解析时，可以用反斜杠 `\` 转义。下面这些符号都应原样显示，而非触发格式：

- 星号：\*这不是斜体\*
- 下划线：\_这不是斜体\_
- 井号：\# 这不是标题
- 反引号：\` 这不是代码
- 方括号与圆括号：\[这不是链接\]\(no-link\)
- 数字加点：1\. 这不是有序列表
- 竖线：a \| b（在表格外）

## 十六、脚注（扩展语法）

脚注允许在正文中插入引用标记，而把注释内容放到文档末尾。正文中这样写：Markdown 最初由 John Gruber 于 2004 年提出[^gruber]，后来衍生出许多方言[^flavors]。脚注是否生效取决于渲染器是否支持该扩展。

## 十七、定义列表（扩展语法）

定义列表用于「术语 + 解释」的结构，常见写法是术语单独一行，解释行以冒号开头：

Markdown
: 一种轻量级标记语言，用纯文本表达富文本结构。

Tauri
: 一个使用系统 WebView 构建跨平台桌面应用的框架，体积小、内存占用低。

WYSIWYG
: 「所见即所得」，编辑时的样式与最终呈现一致。

## 十八、数学公式（扩展语法）

技术文档常常需要数学公式，常见方案是 LaTeX 语法配合 KaTeX/MathJax。行内公式如 $E = mc^2$，独立公式如下：

$$
\int_{a}^{b} f(x)\,dx = F(b) - F(a)
$$

$$
\sum_{i=1}^{n} i = \frac{n(n+1)}{2}
$$

如果这些公式显示为原始的美元符号与字母，说明当前渲染器未启用数学扩展——这本身也是一个有用的测试结果。

## 十九、Emoji 与特殊字符

表情符号可以直接粘贴 Unicode 字符：🚀 🎉 ✅ ❌ 🐛 📝 💡 🔧 ⚙️ 🖱️。部分渲染器还支持「短代码」写法，例如 :smile: :rocket: :tada:（是否生效取决于是否启用 emoji 扩展）。

特殊排版字符测试：版权 ©、注册商标 ®、商标 ™、度数 25℃、箭头 → ← ↑ ↓ ⇒、破折号 —、省略号 …、不等号 ≠ ≤ ≥、以及中文书名号《Markdown 权威指南》。

## 二十二、边界与极端情况

这一节收集一些容易出错的边界用法，用来「找茬」：

- 空的行内代码：`` 与紧贴的标点`，`。
- 连续多个强调符号：***粗斜体*** 紧接 **粗体** 紧接 *斜体*。
- 列表项里直接跟代码块：
  ```text
  缩进对齐的代码块，位于列表项内部
  ```
- 仅含一个字符的单元格表格：

  | a | b |
  | - | - |
  | 1 | 2 |

- 标题紧跟正文（中间无空行）——有的渲染器要求空行，有的不要求：
### 这是一个标题
紧接着的这一行是正文，测试是否被正确分隔。

- 超长不换行的英文/链接，测试是否会撑破布局：`aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa`
- 中文与行内代码紧贴无空格：使用`flushSave`函数保存、读取`draft.markdown`字段。

## 二十三、渲染结果对照检查清单

## 二十四、总结

如果你从头到尾看到这里，并且大部分语法都渲染正确，那么 MiniMarkdown 的核心渲染能力是相当完善的。建议把「对照模式」与「所见即所得模式」对照来看：前者基于 `marked`，通常对 GFM（表格、删除线、任务列表、自动链接）支持更好；后者基于 TipTap，对部分扩展语法（脚注、定义列表、数学公式、内联 HTML）可能不支持，这属于正常现象，而非缺陷。

把渲染异常的条目记录下来，就能清楚地知道接下来要为编辑器补齐哪些扩展。祝测试顺利！🎉
